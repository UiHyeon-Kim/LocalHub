# AGENTS.md

## Project Stack

이 프로젝트는 다음 기술 스택을 사용한다.

* Frontend: Vue.js
* Backend: FastAPI
* Database: SQLite
* Chatbot and AI: Python

사용자 요청 없이 다른 프레임워크, 데이터베이스 또는 주요 기술 스택으로 변경하지 않는다.

## General Rules

* 기존 프로젝트 구조와 코드 스타일을 따른다.
* 요청받은 범위만 수정한다.
* 관련 없는 코드나 파일은 수정하지 않는다.
* 불필요한 리팩터링이나 추상화를 추가하지 않는다.
* 새로운 라이브러리는 꼭 필요한 경우에만 추가한다.
* 가능한 한 단순하고 이해하기 쉬운 코드로 작성한다.
* 기존 기능이 동작하는 방식을 임의로 변경하지 않는다.

## Project Structure

* 프론트엔드, 백엔드, 챗봇 영역의 책임을 구분한다.
* 각 영역의 기존 폴더 구조를 유지한다.
* 공통 API 요청 및 응답 형식을 임의로 변경하지 않는다.
* API 구조를 변경할 경우 연결된 프론트엔드, 백엔드, 챗봇 코드를 함께 확인한다.
* 다른 영역의 코드를 수정해야 하는 경우 필요한 부분만 최소한으로 변경한다.

## Frontend Rules

* Vue.js의 기존 컴포넌트 구조와 작성 방식을 따른다.
* 화면 관련 로직은 프론트엔드 영역에 작성한다.
* API 주소와 환경별 설정값을 코드에 직접 작성하지 않는다.
* 서버에서 관리해야 하는 비밀값을 프론트엔드에 포함하지 않는다.
* 기존 UI 스타일을 유지하고 요청받지 않은 디자인 변경은 하지 않는다.

## Backend Rules

* FastAPI의 기존 라우터, 서비스, 모델 구조를 따른다.
* API 요청과 응답 형식을 명확하게 유지한다.
* HTTP 상태 코드를 상황에 맞게 사용한다.
* 데이터베이스 접근 코드는 기존 방식에 맞춰 작성한다.
* SQLite 테이블이나 컬럼을 변경할 때 기존 데이터에 미치는 영향을 확인한다.
* API Key, 비밀번호, 토큰 등의 비밀값을 응답에 포함하지 않는다.

## Chatbot Rules

* 챗봇 관련 코드는 Python으로 작성한다.
* 프롬프트와 챗봇 처리 로직은 기존 구조를 따른다.
* 챗봇 응답 형식을 임의로 변경하지 않는다.
* 외부 AI API 호출에 필요한 키는 환경 변수로 관리한다.
* 사용자 입력을 프롬프트나 쿼리에 사용할 때 필요한 검증을 수행한다.
* API 호출 실패 시 서버 전체가 중단되지 않도록 기본적인 예외 처리를 한다.

## Environment Variables

* API Key, 비밀번호, 토큰, 데이터베이스 접속 정보 등 민감한 값은 `.env`에 작성한다.
* 실제 값이 포함된 `.env` 파일은 Git에 커밋하지 않는다.
* 필요한 환경 변수 이름은 `.env.example`에 작성한다.
* `.env.example`에는 실제 비밀값을 작성하지 않는다.
* 새로운 환경 변수를 추가하면 `.env.example`도 함께 수정한다.
* 프론트엔드 환경 변수는 사용자 브라우저에 노출될 수 있음을 고려한다.

예시:

```env
# .env.example

DATABASE_URL=
AI_API_KEY=
SECRET_KEY=
```

## Git Rules

* 하나의 커밋에는 하나의 작업만 포함한다.
* 관련 없는 변경을 같은 커밋에 포함하지 않는다.
* `.env`와 민감한 정보가 포함된 파일은 커밋하지 않는다.
* 사용자 요청 없이 커밋, 푸시, 강제 푸시를 수행하지 않는다.

## Commit Message Convention

커밋 메시지는 다음 형식을 사용한다.

```text
<type>: <한글 제목>
```

변경 영역을 표시할 필요가 있는 경우 scope를 추가한다.

```text
<type>(<scope>): <한글 제목>
```

### Commit Types

* `feat`: 새로운 기능 추가
* `fix`: 버그 수정
* `refactor`: 기능 변경 없는 코드 구조 개선
* `chore`: 설정, 환경 구성, 패키지 관리
* `docs`: 문서 작성 또는 수정
* `style`: 동작에 영향을 주지 않는 코드 형식 수정
* `perf`: 성능 개선
* `build`: 빌드 또는 의존성 변경
* `ci`: CI/CD 설정 변경
* `revert`: 이전 변경 사항 되돌리기

### Commit Message Rules

* type과 scope는 영어로 작성한다.
* 커밋 제목은 한글로 작성한다.
* 제목 끝에 마침표를 붙이지 않는다.
* 변경 내용을 한 줄로 간단하고 구체적으로 작성한다.
* `수정`, `작업`, `변경`처럼 의미가 불명확한 제목만 사용하지 않는다.

좋은 예시:

```text
feat(frontend): 장소 검색 화면 추가
feat(backend): 주변 장소 조회 API 추가
feat(chatbot): 장소 추천 응답 기능 추가
fix(backend): 빈 검색어 요청 오류 처리
refactor(chatbot): 프롬프트 생성 로직 분리
chore: 환경 변수 예시 파일 추가
```

피해야 할 예시:

```text
fix: 수정
chore: 작업
feat: 기능 추가 및 버그 수정
update code
최종 수정
```

## Agent Response Rules

* 요청한 작업만 수행한다.
* 변경한 파일과 핵심 내용만 간단하게 설명한다.
* 긴 분석이나 불필요한 코드 품질 설명은 생략한다.
* 별도의 문제가 보여도 요청 없이 함께 수정하지 않는다.
* 작업하지 못한 부분이 있는 경우에만 이유를 알린다.
